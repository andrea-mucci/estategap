# Research: E2E Test Suite (031-e2e-test-suite)

**Branch**: `031-e2e-test-suite` | **Date**: 2026-04-17

All NEEDS CLARIFICATION items from the spec were resolved by design decisions in the user's `/speckit.plan` input. The following research addresses the six open technical questions identified during plan authoring.

---

## Decision 1: pytest-asyncio mode

**Decision**: `asyncio_mode = "auto"` in `pytest.ini` / `pyproject.toml`

**Rationale**: Every existing Python service in the monorepo (ai-chat, ml, spider-workers, pipeline) uses `asyncio_mode = "auto"`. This mode automatically manages the event loop per test function, allowing session-scoped sync fixtures (loading `tests/fixtures/users.json`) to coexist cleanly with `async def` test functions using `httpx.AsyncClient` and `websockets`. No event-loop closure errors occur when mixing fixture scopes. The E2E suite is stateless from the server's perspective (no persistent local state), so the looser lifecycle control of "auto" mode is an acceptable trade-off.

**Alternatives considered**:
- **strict mode**: Requires manually wrapping async session fixtures; adds boilerplate with no benefit for stateless HTTP/WS tests.
- **Legacy mode**: Deprecated; not used in new projects.

## Decision 2: Playwright storageState auth sharing

**Decision**: Pre-authenticate one context per tier in `playwright.config.ts` `globalSetup`, serialize to `frontend/tests/e2e/auth/<tier>.json` (gitignored), inject via `fixtures/auth.ts` into each spec file.

**Rationale**: A `globalSetup` hook runs `POST /api/v1/auth/login` for each test tier (free, basic, pro, admin) once per run, then calls `page.context().storageState({ path })` to persist cookies and localStorage. Each spec file receives a pre-authenticated browser context via a shared fixture, avoiding 30+ redundant login flows per run (each login takes ~300–800 ms; the saving is significant at scale). Logout-flow tests use a separate context without `storageState`. Auth JSON files are regenerated by CI globalSetup, never committed.

**Alternatives considered**:
- **Per-test login**: Simple but adds ~500 ms per test; untenable at 100+ specs.
- **Shared browser context across spec files**: Violates isolation; a logout in one spec poisons the next.

## Decision 3: Rate limit testing strategy

**Decision**: Real HTTP burst within the 1-minute window using `httpx.AsyncClient`; no time mocking; tier tests run in parallel via `pytest-xdist -n 4`.

**Rationale**: The API Gateway uses Redis sliding-window counters with a 60-second TTL. Client-side time mocking (freezegun) has no effect on server-side Redis TTLs. The burst approach — send 35 requests for a Free-tier user (limit 30/min), assert the first 30 return `200`, the 31st returns `429` with a valid `Retry-After` header — executes in under 200 ms using async HTTP. Each tier's test user has a unique identity and thus an isolated Redis key; tests for Free, Basic, Pro, Global, API tiers run in parallel without interference. No sleep required.

**Alternatives considered**:
- **Mock server-side clock**: Requires control of the running server's time syscall; not feasible against a deployed kind cluster.
- **60-second window resets**: Adds 60 s per tier; multiplied across 5 tiers = 5 min overhead. Unacceptable.

## Decision 4: WebSocket 30-minute idle timeout test

**Decision**: Expose `WS_IDLE_TIMEOUT_SECS` environment variable in the WS server; set to `30` in `helm/estategap/values-test.yaml`. Test connects, idles 31 s, asserts `1001` close code.

**Rationale**: A 30-minute real wait is impractical in CI. Patching the server's time is invasive and fragile. An env-var override (`WS_IDLE_TIMEOUT_SECS`, defaulting to `1800`) is the minimal, clean solution: the test config sets it to 30 s, the actual idle-detection code path is exercised identically, and the test completes in ~31 s. The `values-test.yaml` already exists for other test overrides (confirmed in `mk/kind.mk: kind-deploy`). The keepalive ping/pong test runs at the overridden interval too (if `WS_PING_INTERVAL_SECS` is similarly exposed).

**Alternatives considered**:
- **Server-side clock mock**: Invasive; requires patching `time.Now()` in Go WS server.
- **Skip in fast suite, run nightly only**: Acceptable temporary measure; long-term the env-var approach is correct.

## Decision 5: Visual regression baseline strategy

**Decision**: Generate baselines locally via `playwright test visual/ --update-snapshots`, commit PNG files to `frontend/tests/e2e/visual/baselines/{browser}/`. CI always compares against committed baselines at 0.1% threshold.

**Rationale**: Playwright's `toHaveScreenshot()` compares against committed PNG baselines — these must be stable and deterministic. Baseline generation (`--update-snapshots`) is a manual, developer-approved action; running it automatically in CI would make every run trivially pass. Storing baselines in Git (not a cloud bucket) keeps the comparison deterministic and avoids external service dependencies. PNG files are typically 50–200 KB; for 5 pages × 3 browsers = 15 PNGs, total storage is ~3 MB — acceptable in a monorepo. Platform-specific rendering differences are avoided by restricting visual regression to Chromium only (most deterministic cross-platform).

**Alternatives considered**:
- **Generate baselines in CI on every run**: Defeats the purpose; always passes.
- **External service (Percy, Chromatic)**: Additional cost, external dependency, credential management overhead.
- **All three browsers for visual regression**: Cross-platform anti-aliasing variance causes high false-positive rates; Chromium only is the practical choice.

## Decision 6: NATS event injection for deal_alert test

**Decision**: `tests/e2e/helpers/nats_injector.py` uses `nats-py` to publish a `ScoredListingEvent` JSON payload to the `scored.listings` JetStream subject on `localhost:4222`. The real NATS instance is exposed via `tests/kind/port-forward.sh`.

**Rationale**: The WS server subscribes to `scored.listings` (from the NATS consumer in `services/ws-server/internal/nats/consumer.go`) and pushes matching `deal_alert` messages to connected clients whose alert rules match. Publishing via `nats-py` exercises the full production code path — no mocking. The test creates a user with a known alert rule, connects a WebSocket, publishes a scored listing that matches the rule, and asserts a `deal_alert` message arrives within 5 s. NATS is already port-forwarded during `make kind-deploy` (confirmed by `tests/kind/port-forward.sh`). `nats-py` is already used in `services/spider-workers` and `services/pipeline`, so it is an existing transitive dependency.

**Alternatives considered**:
- **Mock NATS in tests**: Reduces confidence in the real alert path; the production trigger is NATS, so the test should use NATS.
- **HTTP trigger endpoint**: No such endpoint exists; would require adding one purely for testing (worse design).
- **Import WS server internal packages**: Violates the constraint that tests do not import service internals.
