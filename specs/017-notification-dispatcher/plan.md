# Implementation Plan: Notification Dispatcher

## Summary

Implement the `alert-dispatcher` Go service that consumes `NotificationEvent` JSON messages from the `alerts.notifications.>` NATS JetStream subject and delivers them through five channels: email (AWS SES with localized HTML templates), Telegram (sendPhoto + inline keyboard), WhatsApp (Twilio template), push (Firebase FCM), and webhook (HTTP POST + HMAC-SHA256). Each channel is implemented as a `Sender` interface. Concurrent multi-channel dispatch uses `errgroup`. Failed deliveries retry up to 3 times (1s/4s/16s backoff); webhook retry state is persisted in Redis. All delivery attempts are recorded in `alert_history`. A new Alembic migration adds dispatcher-specific columns to `users` and `alert_history`.

## Technical Context

**Language/Version**: Go 1.23  
**Primary Dependencies**: nats.go v1.37, go-chi/chi v5, pgx/v5, go-redis/v9, aws-sdk-go-v2/ses, go-telegram-bot-api v5, twilio-go, firebase-admin-go v4, prometheus/client_golang  
**Storage**: PostgreSQL 16 (`users`, `alert_history`, `alert_rules`); Redis 7 (webhook retry counters)  
**Testing**: Go `testing` (table-driven unit tests); testcontainers-go for integration  
**Target Platform**: Linux container (Kubernetes)  
**Performance Goals**: ≥100 notifications/s dispatched; ≤30s end-to-end latency  
**Constraints**: Max 3 retry attempts per channel; webhook retry window ≤300s (Redis TTL)  
**Scale/Scope**: Single service; horizontally scalable via NATS consumer group

## Implementation Phases

### Phase A — Foundation

**Objective**: Service boots, connects to NATS/DB/Redis, and consumes messages without dropping them.

1. **Migration `019_notification_dispatcher.py`**
   - Add to `users`: `preferred_language VARCHAR(10) DEFAULT 'en'`, `telegram_chat_id BIGINT NULL`, `telegram_link_token VARCHAR(64) NULL`, `push_subscription_json TEXT NULL`, `webhook_secret VARCHAR(64) NULL`, `phone_e164 VARCHAR(20) NULL`
   - Add to `alert_history`: `event_id UUID NULL`, `attempt_count SMALLINT DEFAULT 1`
   - Add unique constraint `uq_alert_history_event_channel (event_id, channel) WHERE event_id IS NOT NULL`

2. **`internal/config/config.go`**
   - All env vars; fail fast on missing `DATABASE_URL` or `NATS_URL`
   - Parse `WORKER_POOL_SIZE` (default `runtime.NumCPU() * 4`) and `BATCH_SIZE` (default 50)

3. **`internal/model/types.go`**
   - `NotificationEvent` (copied from alert-engine — no cross-module import)
   - `UserChannelProfile`, `DeliveryResult`

4. **`internal/consumer/consumer.go`**
   - Durable pull consumer: `ALERTS` stream, filter `alerts.notifications.>`, consumer name `alert-dispatcher`, `MaxDeliver=1`, `AckWait=60s`
   - Fetch loop → bounded goroutine pool → ack on success, nack on unrecoverable error

5. **`internal/router/router.go`** + **`internal/metrics/metrics.go`**

6. **`cmd/main.go`** — wire and start

---

### Phase B — Channel Senders

**Objective**: All five `Sender` implementations with in-process retry.

7. **`internal/sender/sender.go`** — `Sender` interface + `withRetry` helper (delays: 1s, 4s, 16s, max 3 retries)

8. **`internal/sender/email.go`**
   - Embed templates with `//go:embed templates/*.html`; parse once at startup
   - Build `EmailTemplateData`; call `ses.SendEmail`; retry on SES transient errors

9. **`internal/sender/telegram.go`**
   - `tgbotapi.NewBotAPI`; `SendPhotoConfig` + `InlineKeyboardMarkup`
   - Long-poll bot goroutine handles `/start {token}` → `repo.StoreTelegramChatID`

10. **`internal/sender/whatsapp.go`**
    - `twilio.NewRestClient`; `CreateMessage` with `ContentSid` + `ContentVariables`

11. **`internal/sender/push.go`**
    - `firebase.NewApp`; `messaging.Send`; on `registration-token-not-registered` → `repo.ClearPushToken` + no retry

12. **`internal/sender/webhook.go`**
    - 10s timeout `http.Client`; HMAC-SHA256 `X-Webhook-Signature`
    - Redis INCR `webhook:retry:{event_id}` (TTL 300s); no retry on 4xx

---

### Phase C — Dispatcher & Repository

13. **`internal/dispatcher/dispatcher.go`**
    - Switch on `event.Channel` → Sender
    - Load `UserChannelProfile` from DB
    - Pre-insert `alert_history` (status `pending`)
    - Call sender; update record to `sent`/`failed`; emit metrics

14. **`internal/repository/history.go`**
    - `Insert`: `ON CONFLICT (event_id, channel) WHERE event_id IS NOT NULL DO NOTHING`
    - `UpdateStatus`: set `delivery_status`, `error_detail`, `attempt_count`, `delivered_at`

15. **`internal/repository/user.go`**
    - `GetChannelProfile(ctx, userID) (*UserChannelProfile, error)`
    - `StoreTelegramChatID(ctx, userID string, chatID int64) error`
    - `ClearPushToken(ctx, userID string) error`

## Key Interfaces

```go
// Sender — implemented by email, telegram, whatsapp, push, webhook
type Sender interface {
    Send(ctx context.Context, event model.NotificationEvent, user *model.UserChannelProfile) (DeliveryResult, error)
}

type DeliveryResult struct {
    Success      bool
    AttemptCount int
    ErrorDetail  string
    DeliveredAt  *time.Time
}

// Retry helper (shared across all senders)
func withRetry(ctx context.Context, maxAttempts int, delays []time.Duration, fn func() error) error
```
