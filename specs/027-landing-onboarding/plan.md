# Implementation Plan: Landing Page & User Onboarding

**Branch**: `027-landing-onboarding` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/027-landing-onboarding/spec.md`

## Summary

Build a statically-generated multilingual marketing landing page at `/{locale}/` (replacing the current chat hero for unauthenticated visitors) and a 3-step interactive onboarding tour (driver.js) shown to new users after registration. Requires a database migration to add `onboarding_completed` to users, an API extension to PATCH /auth/me, and new frontend components for the landing sections and tour controller.

## Technical Context

**Language/Version**: TypeScript 5.5 / Node.js 22 (frontend); Go 1.23 (API Gateway); Python 3.12 (Alembic migration)
**Primary Dependencies**:
- Frontend: Next.js 15 (App Router), next-intl 3.x, driver.js 1.x (new), shadcn/ui accordion + table (new), Tailwind CSS 4, Zustand 5, TanStack Query v5, react-hook-form + Zod
- Backend (Go): chi v5, pgx/v5, existing auth handler patterns
- Migration: Alembic 1.13+, SQLAlchemy 2.0, asyncpg 0.29+

**Storage**: PostgreSQL 16 (users table extension); no Redis usage for this feature
**Testing**: Frontend: Vitest + React Testing Library; Go: table-driven tests; E2E: manual Lighthouse audit
**Target Platform**: Next.js static export (landing page) + server-side components (authenticated app)
**Performance Goals**: Lighthouse Performance > 90, page load < 2s on 3G
**Constraints**: Landing page must be fully statically generated (`force-static`); no dynamic server rendering per-request; onboarding tour must be client-side only (driver.js uses `document`)
**Scale/Scope**: 10 locales × 1 static page = 10 pre-rendered HTML files; onboarding affects all new registrations

## Implementation Phases

### Phase A: Backend Foundation

**Goal**: Add `onboarding_completed` support to the data layer and API before frontend depends on it.

**Tasks**:

1. **DB Migration** — `services/pipeline/alembic/versions/027_add_onboarding_completed.py`
   - `upgrade()`: `op.add_column('users', sa.Column('onboarding_completed', sa.Boolean(), nullable=False, server_default='false'))`
   - `downgrade()`: `op.drop_column('users', 'onboarding_completed')`

2. **SQLAlchemy Model** — `services/pipeline/src/pipeline/db/models.py`
   - Add: `onboarding_completed: Mapped[bool] = mapped_column(Boolean, default=False, nullable=False)`

3. **Go User Struct** — `libs/pkg/models/user.go`
   - Add field: `OnboardingCompleted bool \`json:"onboarding_completed" db:"onboarding_completed"\``

4. **Go Repository** — `services/api-gateway/internal/repository/users.go`
   - Add method: `UpdateOnboardingCompleted(ctx, userID, completed bool) error`
   - SQL: `UPDATE users SET onboarding_completed = $1, updated_at = NOW() WHERE id = $2 AND deleted_at IS NULL`

5. **Go Handler** — `services/api-gateway/internal/handler/auth.go`
   - Extend `PatchMe` request struct: add `OnboardingCompleted *bool \`json:"onboarding_completed"\``
   - After currency update block: if `req.OnboardingCompleted != nil`, call `UpdateOnboardingCompleted`
   - Include `onboarding_completed` in `userProfilePayload` response struct

6. **OpenAPI Spec** — `services/api-gateway/openapi.yaml`
   - Add `onboarding_completed: boolean` to `UserProfile` schema (required)
   - Add `preferred_currency: string` to `UserProfile` schema (was missing)
   - Document `PATCH /api/v1/auth/me` with `UpdateUserProfileRequest` schema (see `contracts/openapi-patch.yaml`)

### Phase D: Onboarding Tour

**Goal**: Build the driver.js tour controller and integrate it into the authenticated app.

**Tasks**:

24. **useOnboarding Hook** — `frontend/src/hooks/useOnboarding.ts`
    - Reads `session.user.onboarding_completed` from NextAuth session
    - On mount: if `!onboarding_completed` → calls `onboardingStore.startTour()`
    - Exposes: `{ active, currentStep, advance, skip }`
    - `skip()`: optimistically calls `PATCH /api/v1/auth/me { onboarding_completed: true }` via existing `updateCurrentUser`, then `onboardingStore.skipTour()`

25. **OnboardingTour Component** — `frontend/src/components/onboarding/OnboardingTour.tsx`
    - Client component (`'use client'`)
    - Reads from `onboardingStore`
    - Lazy-imports driver.js inside `useEffect` to avoid SSR:
      ```ts
      const { driver } = await import('driver.js');
      await import('driver.js/dist/driver.css');
      ```
    - Step 1 (CHAT): Targets `#chat-input` element on `/home`; popover: "Describe what you're looking for"
    - After Step 1 user sends message: `advance(chatCriteria)` → router.push(`/alerts?...criteria`)
    - Step 2 (ALERT): Targets `#alert-form` on `/alerts`; popover: "Set up your first alert"
    - After Step 2 save/skip: `advance()` → router.push(`/dashboard`)
    - Step 3 (DASHBOARD): Targets `#dashboard-summary-card`; multi-step highlight of key cards
    - After Step 3: `completeTour()` → show UpgradeModal

26. **UpgradeModal Component** — `frontend/src/components/onboarding/UpgradeModal.tsx`
    - Uses shadcn `<Dialog>` (already installed)
    - Controlled by `onboardingStore.showUpgradeModal`
    - Content: 3 tiers (Free, Pro, Global) condensed comparison
    - CTAs: "Stay on Free" (close), "Upgrade to Pro" (`/register?tier=pro`), "Get Global" (`/register?tier=global`)
    - On close: calls `updateCurrentUser({ onboarding_completed: true })`

27. **Wire OnboardingTour into Protected Layout** — `frontend/src/app/[locale]/(protected)/layout.tsx`
    - Import and render `<OnboardingTour />` inside the protected layout
    - It is a no-op if `onboarding_completed = true`

28. **Add ID Anchors for Tour Targets**:
    - `frontend/src/app/[locale]/(protected)/home/page.tsx` → add `id="chat-input"` to the chat input wrapper
    - `frontend/src/app/[locale]/(protected)/alerts/page.tsx` → add `id="alert-form"` to the alert creation form; read query params for pre-fill
    - `frontend/src/app/[locale]/(protected)/dashboard/page.tsx` → add `id="dashboard-summary-card"` to the first summary card

29. **Pre-filled Alert Form** — `frontend/src/app/[locale]/(protected)/alerts/page.tsx`
    - Read `useSearchParams()` for `country`, `maxPrice`, `minSize`, etc.
    - Pass as `defaultValues` to react-hook-form `useForm()`
    - Scroll to `#alert-form` on mount if query params present
