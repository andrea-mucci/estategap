# Implementation Plan: ML Training Pipeline

**Branch**: `014-ml-training-pipeline` | **Date**: 2026-04-17 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/014-ml-training-pipeline/spec.md`

---

## Summary

Build a fully automated weekly ML training pipeline inside `services/ml/` that exports listing data from PostgreSQL, engineers ~36 numeric features using a scikit-learn-compatible `FeatureEngineer`, tunes a LightGBM model with Optuna (50 trials), exports the winner to ONNX, and promotes it to active status in `model_versions` only if it improves national MAPE by ≥ 2% over the current champion. Separate per-country models are trained for markets with > 5,000 listings; smaller markets get transfer learning from the Spain base model via LightGBM `init_model`. A Kubernetes CronJob triggers the pipeline every Sunday at 03:00 UTC.

## Phase 1: Design

### FeatureEngineer Design

Sklearn `BaseEstimator + TransformerMixin` so it composes into a `Pipeline` that `skl2onnx` can traverse.

**`fit(df)` responsibilities**:
1. Fetch zone stats per `zone_id` (passed as pre-loaded `dict` from DB query)
2. Compute training-set medians/modes for all numeric/categorical imputers
3. Fit `ColumnTransformer`:
   - `SimpleImputer(strategy="median")` for all numeric nullable columns
   - `SimpleImputer(strategy="most_frequent")` for categorical nullable columns
   - `OrdinalEncoder(categories=[['G','F','E','D','C','B','A']])` for `energy_cert` → 1–7, unknown → 0
   - `OrdinalEncoder(categories=[['to_renovate','renovate','good','new']])` for `condition` → 1–4, unknown → 0
   - `OneHotEncoder(sparse_output=False, handle_unknown='ignore')` for `property_type`
4. Store `feature_names_out_` list

**`transform(df)` responsibilities**:
1. Resolve zone stats per row (3-level fallback)
2. Compute derived features: `usable_built_ratio`, `price_per_m2_eur`, `month_sin`, `month_cos`, `data_completeness`, `has_energy_cert`, `has_photos`, `building_age_years`
3. Apply `ColumnTransformer`
4. Cast to `float32`
5. `assert not np.any(np.isnan(X)) and not np.any(np.isinf(X))`

**Serialisation**: `joblib.dump(fe, "version_tag_feature_engineer.joblib")`. Uploaded alongside ONNX to MinIO.

### Training Loop Design

```
run_training(country)
├── export_training_data()          asyncpg SQL, returns pd.DataFrame (~50k–500k rows)
├── stratified_split()              70/15/15 by city; cities <50 rows → "other" stratum
├── load_zone_stats()               single SQL query → dict[UUID, ZoneStats]
├── FeatureEngineer.fit_transform() on train_df
├── FeatureEngineer.transform()     on val_df, test_df
├── optuna.create_study()
│   └── objective(trial)
│       ├── suggest hyperparams
│       ├── lgb.cv() with early stopping (50 rounds)
│       └── return val MAPE
├── retrain on train+val with best_params (no early stop, fixed n_estimators)
├── evaluate() on test_df           MAE, MAPE, R² + per-city breakdown
├── export_onnx()                   skl2onnx + ONNX self-test
├── maybe_promote()                 asyncpg transaction: champion → retired, new → active
├── log_to_mlflow()                 params + metrics + artefacts + feature importance PNG
└── publish_nats_event()            ml.training.completed or ml.training.failed
```

**Transfer path** (country with < 5k listings):
- Same flow, but skip Optuna; use fixed `{lr: 0.01, n_estimators: 200}`
- Pass Spain booster as `init_model` to `lgb.train()`
- Evaluate, compare, promote independently from Spain model

**Country loop order**: Spain ("es") is always trained first. Other countries follow in listing-count descending order.

### Promotion Transaction Design

```python
async with conn.transaction():
    # 1. Lock current champion row
    champion = await conn.fetchrow(
        "SELECT id, metrics FROM model_versions "
        "WHERE country_code=$1 AND status='active' FOR UPDATE",
        country,
    )
    # 2. Upload artefacts to MinIO (outside transaction to avoid long locks)
    #    → do this before opening transaction
    # 3. Insert new model_versions row (status='staging')
    new_id = await conn.fetchval("INSERT INTO model_versions (...) VALUES (...) RETURNING id")
    # 4. If champion exists and threshold not met → update new row to 'retired', return False
    # 5. If promoting:
    await conn.execute("UPDATE model_versions SET status='retired', retired_at=NOW() WHERE id=$1", champion_id)
    await conn.execute("UPDATE model_versions SET status='active', promoted_at=NOW() WHERE id=$1", new_id)
```

MinIO upload happens *before* the DB transaction to keep transaction time short. If MinIO upload fails, the DB row is never inserted (the exception propagates).

### Kubernetes CronJob

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: ml-trainer
  namespace: {{ .Release.Namespace }}
  labels: {{ include "estategap.labels" . | nindent 4 }}
spec:
  schedule: "0 3 * * 0"
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      backoffLimit: 0
      template:
        spec:
          restartPolicy: Never
          serviceAccountName: ml-trainer
          containers:
            - name: ml-trainer
              image: "{{ .Values.mlTrainer.image.repository }}:{{ .Values.mlTrainer.image.tag }}"
              command: ["python", "-m", "estategap_ml.trainer", "--countries-all"]
              resources:
                requests:
                  memory: "4Gi"
                  cpu: "2"
                limits:
                  memory: "8Gi"
                  cpu: "4"
              envFrom:
                - secretRef:
                    name: ml-trainer-secrets
                - configMapRef:
                    name: estategap-config
```

`--countries-all` flag reads distinct `country_code` values from `listings` and runs the full loop. On failure of any country: publish `ml.training.failed`, continue to next country. If Spain fails: skip all transfer-learning countries (they depend on Spain booster).
